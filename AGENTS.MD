# AGENTS.md — Repository Instructions

Engineering assistant guidelines for maintaining, testing, and extending this codebase cleanly and safely.

---

## 1. Operating Principles & Workflow

- **Inspect First:** Review repo structure, module boundaries, dependencies, and existing patterns before making changes.
- **Minimal & Reversible Scope:** Prefer small, focused, incremental changes over broad refactoring or framework additions. Do not optimize for speed over correctness.
- **Layering & Architecture:** Respect established domain boundaries. Never bypass layers (e.g., API -> DB directly is strictly forbidden; follow API -> Business Logic -> Data Access).
- **Dependencies:** Verify if existing libraries solve the issue before proposing new dependencies. Check compatibility, maintenance, and security.
- **Feature Completion:** A feature is incomplete without matching tests and updated docs.

---

## 2. Safety, Data & Security Rules

- **Zero Secrets:** Never commit credentials, API keys, tokens, or sensitive configuration files.
- **Database Integrity:** Handle persistence with migrations, explicit transactions, index considerations, and rollback strategies. **Never make destructive data changes without explicit user confirmation.**
- **API Contracts:** Maintain backward compatibility, input validation, and proper error responses. Avoid breaking existing clients.

---

## 3. Quality, Testing & Debugging

- **Testing Standard:** Always run existing tests, add regression coverage for bugs, and write behavior/integration tests for new behavior. Avoid testing internal implementation details.
- **Debugging Protocol:** Reproduce $\rightarrow$ Inspect logs $\rightarrow$ Locate root cause $\rightarrow$ Apply minimal fix $\rightarrow$ Add regression test. Never hide errors or add unverified workarounds.

---

## 4. Documentation & Git Commits

- **Commits:** Write clear, human-style conventional commits (e.g., `feat(scope): ...`, `fix(scope): ...`).
    - **Forbidden:** Never include AI signatures (`Generated by AI`, `Created by Claude`, etc.).
    - **Format Check:** Ensure messages complete the phrase *"If applied, this commit will..."*
- **Documentation:** Update architectural notes, APIs, setup steps, or workflows whenever relevant. Explain *what* changed and *why*.

---

## 5. Execution Rules & Completion Checklist

### DO
- Ask questions if requirements, security, or business logic are ambiguous—do not guess.
- Re-use established patterns and explain significant decisions.

### DO NOT
- Invent requirements, substitute working modules without reason, or ignore failing tests.

### Checklist Before Completion
- [ ] Structure and patterns understood and respected.
- [ ] Smallest viable solution implemented.
- [ ] Tests added/updated and executed successfully.
- [ ] Zero secret leakage and database migration safety checked.
- [ ] Docs updated and changes cleanly summarized without bot signatures.
